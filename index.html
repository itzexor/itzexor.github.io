01

Guiding Principles

Single peer dependency

Three.js only. No React, no Vue, no build system assumptions. Works anywhere a canvas element works.

Environment-aware, not environment-dependent

Inject a slicer environment and the plugin drives itself. Omit it and the user configures manually. Same API either way.

Plugin owns geometry only

Spatial reasoning, transforms, overlap detection, the canvas. Everything else — file pickers, filament databases, printer management — belongs to the host shell.

Slots, not filaments

The plugin thinks in slot numbers. The host maps those to real filament profiles. The plugin carries that data opaquely into 3MF output.

mm everywhere

All public API coordinates, dimensions, and transforms are in millimeters. Three.js scene units are an internal implementation detail.

Headless capable

All geometry, arrangement, and export logic runs without a canvas. Useful for server-side pre-arrangement, CI testing, and CLI tooling.

Replaceable algorithms

Auto-arrange, collision, and bin-packing are pluggable. The host can swap in its own strategy via a clean function interface.

Opt-in UI

The built-in config panel is optional and unstyled by default. The host can ignore it entirely and build their own shell UI using the same events.

03

Environment Context

The host slicer shell injects an environment object at init time. The plugin uses this to auto-configure bed dimensions, material slots, plate limits, and export profile data — replacing all manual configuration. If no environment is provided the plugin falls back to manual config mode.

Structure

const plater = new Plater(canvasEl, {
  env: {
    printer: {
      id:        'bambu-x1c',
      name:      'Bambu X1C',
      bed:       { x: 256, y: 256, shape: 'rect' }, // mm
      maxPlates: 16,
      extruders: 1,
      amsSlots:  4,
    },
    materials: [
      { slot: 1, id: 'bambu-pla-red',   label: 'PLA Basic Red',   color: '#e74c3c' },
      { slot: 2, id: 'bambu-pla-white', label: 'PLA Basic White', color: '#ecf0f1' },
      { slot: 3, id: 'bambu-pla-black', label: 'PLA Basic Black', color: '#2c3e50' },
    ],
    // Carried opaquely into 3MF extensions — plugin never interprets this
    slicerProfile: { ... },
  }
});

Runtime Updates

The slicer shell calls updateEnv when the user changes printer, swaps filament, or adds an AMS unit. The plugin diffs the new env against the current state and reacts accordingly.

plater.updateEnv({
  materials: [ ...updatedList ],    // re-renders slot colors on parts
  printer: { amsSlots: 8 },        // expands available slot count
});

Conflict Detection

If an env update removes a slot that parts are currently assigned to, the plugin fires env:conflict rather than silently breaking. The host decides how to surface this to the user.

Graceful Degradation

Plate Management

Each plate is an independent layout context sharing the same bed dimensions. Auto-arrange fills plates in order, overflowing to the next. The host can pre-define a fixed count or let the plugin grow plates automatically.

Configuration

plater.setBedConfig({
  x:          256,          // mm — ignored if env.printer.bed is set
  y:          256,
  shape:      'rect',      // 'rect' | 'circle' (delta printers)
  plateCount: 'auto',     // 'auto' | number
  spacing:    2,           // mm between parts
});

05

Part Management

Parts are the primary objects. Each has a mesh, transform, material slot assignment, and metadata. Quantities spawn multiple independent instances that can be arranged separately.

Adding Parts

// Returns id (single) or id[] (quantity > 1)
const id = await plater.addPart(arrayBuffer, {
  id?:           'leg',       // auto-generated if omitted
  name?:         'Chair Leg',
  format?:       'stl',       // 'stl' | '3mf' — auto-detected if omitted
  quantity?:     4,           // spawns 4 independent instances
  materialSlot?: 1,
  plateId?:      'plate-1',  // target plate, defaults to active
  color?:        '#hex',     // override slot color for this part
  locked?:       false,      // prevent drag/rotate
});

Transform API

All positions in mm. Rotation is Z-axis only for standard FDM placement. Flip and orient helpers handle common pre-print operations.

plater.movePart(id, { x, y })          // absolute mm position on plate
plater.rotatePart(id, degrees)          // Z-axis, 0–360
plater.flipPart(id, 'x')               // mirror on X or Y
plater.orientPart(id, face)             // 'flat' | 'front' | 'back' | 'left' | 'right'
plater.dropToBed(id)                   // settle Z to bed surface
plater.centerPart(id)                  // center on current plate

plater.getTransform(id)               // → { x, y, rotationZ, flipX, flipY }
plater.getPart(id)                     // → { id, name, transform, bounds, slot, overlapping }
plater.getParts(plateId?)              // → Part[] optionally filtered by plate

plater.removePart(id)
plater.setPart(id, { name, locked, color, materialSlot })
plater.clearPlate(plateId?)            // clear active plate or specific plate
06

Material Slots

The plugin operates on slot numbers only. The host maps those numbers to real filament profiles in their shell. The plugin renders parts in the slot's color, tracks assignments, and embeds slot data into 3MF output. Actual filament metadata is injected by the host at export time via onBeforeWrite.

When env is provided

Slots are auto-populated from env.materials. Labels and colors come from the slicer environment. The user never manually defines slots.

When env is absent (manual mode)

plater.setMaterialSlots([
  { slot: 1, label: 'Slot 1', color: '#e74c3c' },
  { slot: 2, label: 'Slot 2', color: '#3498db' },
]);

Assignment API

MethodReturnsDescription
setPartMaterial(id, slot)voidAssign a slot to a part; re-renders color
getPartMaterial(id)numberCurrent slot number for a part
getPartsBySlot(slot)id[]All parts assigned to a slot
getMaterialSummary()SlotSummary[]Per-slot part count and color

Export Hook

const blob = await plater.export3MF({
  onBeforeWrite: (manifest) => {
    // manifest.slots = [{ slot: 1, partIds: [], materialId, color }]
    // host enriches with real filament profile data
    manifest.slots[0].filamentProfile = myDB.lookup('bambu-pla-red');
    return manifest;
  }
});
When env is provided, the slot→material mapping is already known at init time. onBeforeWrite is not needed for the common case — the plugin resolves filament data automatically from env.materials and env.slicerProfile. The hook exists for overrides and edge cases.
07

Interaction

All interaction is constrained to XY — Z movement is not exposed to the user. Parts are dragged on the bed plane, rotated around their Z axis, and selected individually or as a group.

Drag

Raycasting against the bed plane. Parts snap to the bed surface and stay within bounds unless the host disables boundary enforcement. Multi-select drag moves all selected parts together while preserving relative offsets.

Rotation

A gizmo handle appears on the selected part. Dragging it rotates around Z. Holding Shift snaps to 45° increments. Holding Ctrl snaps to 90°.

Selection

Click to select. Shift+click to add to selection. Click empty space to deselect. Box-select by dragging on the bed plane with no part under cursor.

Configuration

plater.setInteractionConfig({
  snapToGrid:    false,
  gridSize:      1,         // mm
  boundaryClamp: true,      // prevent parts leaving bed
  multiSelect:   true,
  rotateHandle:  true,
});

Programmatic Selection

plater.select(id | id[])
plater.deselect(id?)          // omit to deselect all
plater.getSelected()          // → id[]
08

Algorithms

Auto-Arrange

Projects each part's geometry to a 2D XY footprint polygon, then runs a bin-packing algorithm to place all unlocked parts across plates with minimum gaps. Locked parts are treated as fixed obstacles.

await plater.autoArrange({
  strategy: 'pack',   // 'pack' (compact) | 'grid' (uniform rows)
  spacing:  2,        // mm between parts — defaults to bedConfig.spacing
  scope:    'all',    // 'all' | 'selected' | 'plate'
});

Custom Strategy

The host can replace the built-in packing algorithm entirely. The function receives part footprints and bed dimensions and returns transforms.

plater.setArrangeStrategy((parts, bed) => {
  // parts: [{ id, footprint: Polygon2D, width, height }]
  // bed:   { x, y }
  // return: [{ id, x, y, rotation }]
  return myNestingEngine.pack(parts, bed);
});

Collision Detection

Uses convex hull projections rather than axis-aligned bounding boxes to accurately detect overlap between irregular shapes. Computed once on load, cached, and updated only on transform change.

Overflow Handling

When a plate fills, parts that don't fit are automatically moved to the next plate. If plateCount is 'auto' a new plate is created. If a fixed count is set, overflow parts are flagged via the bed:overflow event and left unplaced.

09

Import / Export

Import

// Single geometry file → placed on active plate
await plater.addPart(buffer, options)

// Full 3MF project → restores complete layout, plates, and slot assignments
await plater.import3MF(buffer)

Export

// Full 3MF blob — geometry + transforms + material slots + slicer profile
const blob = await plater.export3MF({
  plates?:       'all' | id[],   // default: all plates
  onBeforeWrite: (manifest) => manifest,
  extensions?:   { bambu: {}, prusa: {} },
})

// One 3MF blob per plate
const blobs = await plater.exportPlates3MF()  // → [{ plateId, blob }]

// Layout JSON only — transforms, no geometry
const json = plater.getLayout()
plater.setLayout(json)   // restore transforms (parts must already be loaded)

3MF Construction

Built client-side using fflate for ZIP construction. No server round-trip required for export. Each part becomes a mesh in 3dmodel.model. Transforms are written as matrix entries. Slot assignments go into object-level metadata. The env.slicerProfile blob is injected into slicer-specific extension namespaces.

11

Optional Config Panel

The plugin ships a minimal, unstyled DOM panel the host can mount into any container element. It is fully optional — hosts that build their own shell UI can ignore it entirely and drive everything through the API. The panel fires the same events as the API so the host reacts identically regardless of which UI is in use.

plater.mountConfigPanel(containerElement, {
  show: [
    'bedSize',       // hidden if env.printer.bed is set
    'plateCount',    // hidden if env.printer.maxPlates is set
    'spacing',
    'materialSlots', // hidden if env.materials is set
    'snapToGrid',
  ],
});

plater.unmountConfigPanel();
When a full env is provided, most config panel fields become redundant — bed size, slot labels, and plate limits are already known. The host should hide the panel or show only fields that remain user-configurable (e.g. spacing, snap-to-grid).

The panel emits config:changed on every user interaction so the host can persist settings or validate against printer constraints before they take effect.

12

Headless Mode

All geometry, arrangement, and export logic works without a canvas. Pass null as the canvas element to run in headless mode. Useful for server-side pre-arrangement, automated plate generation in a build pipeline, and unit testing algorithm behavior without a DOM.

// No canvas — geometry and IO only
const plater = new Plater(null, { env });

await plater.addPart(stlBuffer, { quantity: 4, materialSlot: 1 });
await plater.autoArrange();

const blob = await plater.export3MF();
// → write blob to disk, return from an API route, etc.

In headless mode, all interaction APIs (select, movePart via drag, etc.) are still available programmatically. The Three.js renderer is not instantiated.

Responsibility Boundary

The plugin and host shell have a hard boundary. The plugin never crosses into host territory; the host never needs to know about Three.js internals.

  • Three.js scene, renderer, camera
  • Part geometry, transforms, bounds
  • Overlap and collision detection
  • Bin packing and auto-arrange
  • Material slot numbers and colors
  • Plate layout and overflow logic
  • 3MF construction and serialization
  • Layout JSON serialization
  • Canvas and optional config panel DOM
  • Environment diffing and conflict detection
  • File picker UI
  • Part list, thumbnails, quantity inputs
  • Filament profile databases
  • Printer management and connection
  • Slicer invocation (CLI or API)
  • Print queue management
  • AMS / MMU hardware configuration
  • Settings panels beyond config panel
  • Authentication, cloud sync
  • Any UI outside the canvas element
